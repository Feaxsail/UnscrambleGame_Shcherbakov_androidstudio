# Unscramble Game — Лабораторная работа №13-14

## 📱 Название проекта

**Unscramble Game** — интерактивная игра для отгадывания перемешанных слов на Android.

---

## 🎮 Краткое описание игры

Игрок видит перемешанное слово и должен ввести его правильное написание. За каждый правильный ответ начисляется 20 очков. Игра состоит из 10 слов. После прохождения всех слов показывается диалог с результатами и возможностью начать заново.

### Правила игры:
- Всего **10 слов** для отгадывания
- За каждый правильный ответ: **+20 очков**
- Максимальный счёт: **200 очков**
- Можно **пропустить** слово без потери очков
- При неправильном ответе: показывается красное сообщение об ошибке

## ❓ Контрольные вопросы

### 1. Что такое ViewModel и зачем он нужен?

**ViewModel** — это архитектурный компонент из Android Jetpack, который предназначен для хранения и управления данными UI.

**Какую проблему решает ViewModel:**
- Без ViewModel данные хранятся прямо в Composable-функциях или Activity
- При повороте экрана Activity уничтожается и создаётся заново
- Все данные теряются, пользователь начинает с начала
- ViewModel **переживает** изменение конфигурации (поворот экрана, смену темы)

**Что происходит без ViewModel:**
```kotlin
// ❌ Без ViewModel — данные теряются
var score by remember { mutableStateOf(0) }
// После поворота: score = 0

// ✅ С ViewModel — данные сохраняются
val viewModel: GameViewModel = viewModel()
val score by viewModel.uiState.collectAsState()
// После поворота: score сохраняется!
```

## 2. В чём разница между MutableStateFlow и StateFlow?

| Характеристика | MutableStateFlow | StateFlow |
|----------------|------------------|-----------|
| Изменяемость | Можно изменять | Только чтение |
| Где используется | Внутри ViewModel | В UI (Composable) |
| Модификатор | `private val` | `public val` |

### Зачем создавать два поля `_uiState` и `uiState`:

```kotlin
// Приватное — можно изменять только внутри ViewModel
private val _uiState = MutableStateFlow(GameUiState())

// Публичное — UI может только читать
val uiState: StateFlow<GameUiState> = _uiState.asStateFlow()
```

## 3. Почему в ViewModel нельзя хранить Context или View?

ViewModel не должен содержать Context или View, потому что:

### Что может произойти:

1. **Утечка памяти (Memory Leak)** — ViewModel живёт дольше чем Activity
2. Если ViewModel хранит ссылку на Activity, то Activity не может быть уничтожена
3. Система не может освободить память, даже когда Activity больше не нужна
4. Приложение начинает потреблять больше памяти → crash

### Правильный подход:

```kotlin
// ❌ НЕПРАВИЛЬНО — утечка памяти!
class GameViewModel : ViewModel() {
    private var context: Context = ... // Ссылка на Activity!
}

// ✅ ПРАВИЛЬНО — нет ссылок на UI
class GameViewModel : ViewModel() {
    private val _uiState = MutableStateFlow(GameUiState())
    // Только данные, никаких Context или View
}
```
## 4. Чем StateFlow отличается от remember и rememberSaveable?

| Подход | Сохраняет при повороте | Сохраняет при смерти процесса | Где используется |
|--------|----------------------|------------------------------|------------------|
| `remember` | ❌ Нет | Нет | Временное состояние в Composable |
| `rememberSaveable` | ✅ Да | ✅ Да | Простые данные (строки, числа) |
| `StateFlow + ViewModel` | ✅ Да | ❌ Нет | Логика приложения, сложное состояние |

### Когда лучше использовать каждый подход:

| Ситуация | Рекомендуемый подход |
|----------|---------------------|
| Временное состояние UI (анимация, фокус) | `remember` |
| Ввод пользователя, который нужно сохранить при повороте | `rememberSaveable` |
| Данные игры, логика приложения, состояние экрана | `StateFlow + ViewModel` |
| Данные, которые нужно сохранить после закрытия приложения | `Database / DataStore` |

## 5. Что такое data class в Kotlin и зачем он используется для UI State?

**data class** — это специальный тип класса в Kotlin, предназначенный для хранения данных.

### Автоматически генерируемые методы:

| Метод | Описание |
|-------|----------|
| `equals()` | Сравнение двух объектов |
| `hashCode()` | Вычисление хэш-кода |
| `toString()` | Строковое представление объекта |
| `copy()` | Копирование объекта с изменениями |

### Пример использования `copy()`:

```kotlin
// Исходное состояние
val currentState = GameUiState(score = 0, isGameOver = false)

// Создаём копию с изменённым score
val newState = currentState.copy(score = 20)
// result: GameUiState(score = 20, isGameOver = false)

// Все остальные поля остаются прежними!
```
## 6. Как работает collectAsState() в Compose?

**`collectAsState()`** — это функция-расширение, которая:

- Подписывает Composable на изменения StateFlow
- Автоматически перерисовывает UI при изменении данных
- Преобразует StateFlow в State для использования в Compose

### Почему UI автоматически обновляется:

```kotlin
@Composable
fun GameScreen(viewModel: GameViewModel = viewModel()) {
    // Подписка на StateFlow
    val gameUiState by viewModel.uiState.collectAsState()
    
    // При изменении _uiState.value → UI перерисовывается автоматически!
    Text(text = "Счёт: ${gameUiState.score}")
}
```
## 7. Зачем разделять код на пакеты data и ui?

Разделение на пакеты — это принцип **Separation of Concerns** (Разделение ответственности).

### Структура нашего проекта:

com.shcherbakov.unscramblegame/
├── data/ # Модели данных
│ ├── GameUiState.kt
│ └── wordsData.kt
├── ui_model/ # UI логика
│ ├── GameViewModel.kt
│ └── GameScreen.kt
└── ui/theme/ # Темы и цвета
└── Color.kt


### Какие принципы архитектуры это поддерживает:

| Принцип | Описание |
|---------|----------|
| **Separation of Concerns** | Каждый пакет отвечает за свою задачу |
| **Single Responsibility** | Каждый класс имеет одну ответственность |
| **Modularity** | Код легко тестировать и поддерживать |
| **Scalability** | Легко добавлять новые функции |
| **Reusability** | Компоненты можно использовать в других местах |

### Преимущества:

- ✅ **Легко найти нужный код**
- ✅ **Легко тестировать отдельные компоненты**
- ✅ **Несколько разработчиков могут работать параллельно**
- ✅ **Меньше конфликтов при слиянии кода**
